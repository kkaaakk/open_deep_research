# 项目问题分析与改进建议

> 本文档基于第三方对简历项目的反馈意见，结合 `open_deep_research` 代码仓库的实际情况，逐条分析存在的问题并给出改进方向。

---

## 一、关键前提：明确项目 Ownership

`README.md` 显示作者为 **Lance Martin**，项目挂在 `github.com/langchain-ai` 组织下，有官方 blog、视频课程、HuggingFace 排行榜。如果简历将其作为**个人项目**陈述，面试官会立刻识别出问题。

**简历上必须明确：**
- 你的角色（author / contributor / 二次开发者）
- 你贡献的具体模块
- 哪些是你从零做的，哪些是基于开源项目改的

同理，如果你写的 coding agent 只是 Codex / Claude Code 的 wrapper 或复刻，那它就没有独立存在价值，建议直接删掉。

## 二、RAG 模块的具体问题

### 问题 1：堆砌技术，而非展示判断力

RAG 模块有 **27 个文件**，几乎覆盖了所有可选技术栈：

| 维度 | 选项 |
|------|------|
| Embedding | SentenceTransformers / Hash |
| 向量库 | Milvus / FAISS / Chroma / InMemory |
| 关键词检索 | 内存 BM25 / Elasticsearch BM25 |
| 重排 | Cross-Encoder / KeywordOverlap / NoOp |
| GraphRAG | 内存 / Neo4j |
| 文档加载 | txt / md / json / pdf / png / jpg + OCR + Vision |

这给人的印象是「我把能想到的都实现了一遍」。但面试官要看的是**设计思路和 trade-off 判断**，不是你堆了多少技术。

一个合格的工程师面对这么多选项，会做出明确选择并解释原因：

- **为什么默认选 Milvus 而不是 FAISS？**（Milvus 支持持久化和分布式，FAISS 是纯内存，适合不同场景）
- **为什么选 MiniLM 而不是其他 embedding 模型？**（多语言支持 + 轻量级，在中文场景下的具体表现）
- **为什么 Cross-Encoder 重排是必须的？**（向量召回 + BM25 召回的精度天花板分析，过了哪条线就需要精排介入）
- **BM25 + 向量混合召回的 alpha 是怎么定的？**（0.65 偏向向量，这个值是调出来的还是拍脑袋？）

现有的 `rag.md` 和 `rag-notes.md` 是**参考文档**（描述每个组件做什么），不是**设计文档**（解释为什么做这个选择）。

### 问题 2：RAG 评测有了，但叙事不完整

评测框架本身方向是对的——递增噪声测试 + 分 category 统计 + miss 诊断：

```
记录1 (hash 基线)         → hit@5=0.90, precision@5=0.68
记录2 (真实模型)          → hit@5=1.0,  precision@5=0.73
记录3 (扩充知识库)        → hit@5=1.0,  precision@5=0.71
记录4 (加入误导)          → hit@5=1.0,  precision@5=0.53
记录5 (更多误导)          → hit@5=0.95, precision@5=0.43
```

但问题在于：

1. **只有 5 轮，跨了 2 天**——看起来像突击补的，不像持续迭代的工程实践
2. **只有记录，没有行动**——当 `precision@5` 从 0.71 跌到 0.43 时，你做了什么优化？还是仅仅记录了？
3. **没有 ablation study（消融实验）**——去掉 BM25 会怎样？去掉 reranker 会怎样？去掉 GraphRAG 会怎样？
4. **跨语言问题未被深挖**——`cross_lingual` 的 MRR 在记录 4 中跌到 0.62，这说明中文检索的 reranker 模型可能不够好，但没有进一步分析

反馈里说的**「分析召回率和稳定性」**，如果只有数据没有分析和改进闭环，约等于没有。

### 问题 3：Memory + Knowledge 的自进化叙事缺失

项目**已经有**自进化的基础设施，但没有包装成这个叙事：

- `memory/` 模块：MySQL 持久化对话记忆，5 种 record type（summary, preference, project_fact, decision, constraint, deprecated）
- `rag/memory_writer.py`：对话完成后写入 MySQL
- `rag/mysql_memory.py`：MySQL memory 进入 RAG 索引
- `rag/loaders/mysql_memory.py`：定时将 pending memory 刷新到向量库

这些基础设施可以支撑一个非常好的叙事——**「知识自沉淀和结构化入库」**，但简历/文档里完全没有从这个角度讨论。

应该回答的问题：

- **沉淀策略**：什么内容值得进知识库？怎么避免噪声积累？
- **结构化设计**：为什么用 5 种 record_type？decision 和 project_fact 在检索时权重是否不同？
- **衰减机制**：deprecated 类型的记忆如何影响检索分数？（项目有 authority 系统做这个，但没有总结成方法论）
- **冲突处理**：新记忆与旧知识矛盾时怎么办？覆盖还是共存？

这些才是 agent 工程师要思考的问题，而不是「我用了 Milvus 还是 FAISS」。

### 问题 4：GraphRAG 徒有其表

GraphRAG 的实现是基于**共享实体/术语的共现图扩展**，设计上没有大问题，但它被做成了一个「开关」（`rag_graph_enabled`），而不是一个「有设计意图的子系统」。

反馈里说的**「聊 wiki 和 graph 结构」**，指的是知识的结构化组织方式，不是挂了 Neo4j 就算数。应该讨论：

- 什么时候图扩展有效，什么时候是噪声？（实验数据）
- 图权重 `graph_weight=0.35` 是怎么定出来的？（有没有做参数扫描？）
- 共现图的边是怎么建的？实体识别用的是规则还是模型？
- 与传统知识图谱（entity-relation-entity）的差距在哪里？
- 图结构和 wiki 类知识的索引方式有什么关联？

目前这些都没有展开讨论。

## 三、架构层面的问题

### 问题 5：配置过载

`configuration.py` 有 **~100 个配置项**。这不是灵活，是缺乏设计决断力。

agent 工程师考验的是**「技术经验和产品能力」**。产品能力意味着在灵活性和易用性之间做出取舍：

- 100 个配置项导致用户不知道哪些是关键配置
- 组合爆炸导致无法充分测试
- 配置之间的交互没有文档化
- 很多配置项（如 4 种向量库切换）在实际使用中没有意义——你真的需要在生产环境频繁切换 FAISS 和 Chroma 吗？

好的产品设计会把 80% 的场景收敛到 5-8 个核心配置，剩下的才是高级选项。

### 问题 6：Public Opinion 工作流被埋没

项目中**最有产品价值**的东西是 public opinion（舆情风险监控）工作流：

- 4 个专业 agent（信号采集、内部知识、风险评估、应对策略）
- 有真实的业务场景数据（Tesla 电池危机案例、法规文档、FAQ、PR 手册）
- 有企业微信 / 飞书集成（MCP）
- 有完整的 DAG 编排

这是一个**有明确场景、有真实数据、有清晰价值**的模块。但它被埋在一个庞大的通用框架里，README 甚至没有重点介绍它。

反馈里说的「娱乐级项目 + 开源库」反而更容易约面，是因为它们**场景明确、需求清晰、有真实用户**。Public Opinion 模块有同样的潜质，但被作为配置项的一个枚举值而不是项目的核心叙事来呈现。

## 四、改进建议

### 短期（简历 / 面试层面）

1. **明确你的角色**：
   - 如果你是这个项目的 contributor，说清楚你贡献了哪些模块（RAG 系统？Memory？Public Opinion？）
   - 如果你是基于它做了二次开发，说清楚你加了什么
   - 绝对不要让面试官觉得你在「冒领」开源项目

2. **把 RAG 重新包装为「自进化知识管理模块」**：
   - 核心叙事：知识沉淀 → 结构化入库 → 检索 → 衰减与冲突处理 → 评测闭环
   - 重点讲设计决策，而非技术栈列表
   - **不要再强调你支持了多少种向量库**

3. **突出 Public Opinion 场景**作为独立项目展示：
   - 比通用 deep research 有说服力得多
   - 展示你对垂直场景的理解（舆情监控的业务流程、agent 角色分工的依据）
   - 强调知识库的构建思路（法规、历史案例、PR 手册的结构化方式）

4. **删掉 coding agent 项目**（如果简历上有），除非它是你从零写的且有独特设计理念。

### 中期（项目改进层面）

1. **补消融实验**：
   - 纯向量 vs 向量+BM25 vs 向量+BM25+reranker vs 全链路
   - 每个 category 的 recall/precision/MRR 变化
   - 这就是反馈说的「分析召回率和稳定性」
   - 记录每次优化对指标的定量影响，形成闭环

2. **写设计文档而非参考文档**：每个模块回答三个问题——
   - 为什么做这个？（场景驱动，而非「因为我需要做个 RAG」）
   - 为什么选这个方案？（trade-off 分析，对比备选方案）
   - 实际效果怎样？（量化指标 + 真实使用反馈）

3. **收敛配置**：
   - 把 100 个配置项收敛到 10 个核心配置
   - 其余给出合理默认值
   - 证明你有产品判断力，知道什么该暴露给用户、什么不该

4. **做一个你真正会用的东西**：
   - 反馈里很关键的一句话：**「你真的会用他吗？不用的话怎么进一步测试调优？」**
   - 如果你自己都不用，评测就永远是 mock 数据，永远发现不了真实问题
   - 从自己的日常需求出发，做一个能解决自己实际问题的工具
   - 娱乐项目如果真有人用（哪怕只有几十个 star），比一个没人用的企业级框架有说服力得多

## 五、总结

反馈的核心结论：

> 你的项目有技术堆砌但没有工程判断，有实现但没有设计，有评测但没有迭代。agent 工程师的价值不在于你会用多少工具，而在于你面对一个模糊需求时，能否拆解、架构、取舍、验证。

好消息：项目基础设施不差——RAG 评测框架、memory 系统、authority 机制、舆情工作流都是有价值的。

坏消息：它们被藏在一个「万能框架」里，没有讲述它们存在的理由和设计过程。

**行动方向：把 RAG 重新讲成一个「自进化的知识管理问题」，把舆情工作流讲成一个「真实场景驱动的多 agent 协作问题」。这样你的项目会从一个「复刻开源库的项目」变成一个「展示了 agent 工程能力的项目」。**
